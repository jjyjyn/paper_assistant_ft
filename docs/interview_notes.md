# 面试与答辩笔记（Interview Notes）

> 这是“快读版”。如果你要按环节准备老师追问，先看 `docs/03_interview/teacher_question_bank.md`。

## Why：今天的数据方案为什么这么设计

### 为什么先做“结构化案例库 + 模板扩展”

- 目标是快速形成可训练数据闭环，避免在环境波动期空耗时间。
- 用 20 条结构化案例可稳定扩展到 80 条四任务样本，覆盖面可控。
- 模板方法便于复现和迭代，后续可以按同样框架持续加样本。

### 为什么要四类任务等比例覆盖

- 项目目标不是单一问答，而是“贡献提取 + 对比解释 + 实验解析 + 追问答辩”完整能力。
- 等比例分布可避免模型偏科，训练和评测更容易定位短板。

### 为什么先做 smoke 训练再正式训练

- 降低算力浪费，先验证配置、数据路径、日志链路是否正确。
- 先看显存和吞吐，再调 batch/accum，正式训练更稳。

## Day 2 可能被问

1. 你的数据是不是“模板化太重”？
- 回答：v1 目的是最短时间拿到可用闭环；v2 会补人工高难样本和边界样本，模板是启动手段，不是终态。

2. 为什么不是直接人工写 1000 条？
- 回答：时间和成本不匹配当前阶段目标。先做可复现的中小规模高覆盖集，再按误差驱动扩容更高效。

3. 你如何保证数据质量？
- 回答：通过字段校验、任务分布校验、去重和抽样人工检查；并记录 source_case_id 保障可追溯。

## 关于租 4090 的答辩话术

1. 为什么选 4090？
- 回答：对 3B-4B LoRA 微调足够，单小时成本低，最符合“快速闭环”的工程目标。

2. 你如何控制成本？
- 回答：先本地完成数据与脚本，只在训练窗口租机；先 smoke 再长训，按效果决定是否继续投入时长。

3. 如果训练失败怎么办？
- 回答：先回放日志定位问题，修复后复用同一配置快速重启；数据和脚本已固化，迁移成本低。

## Phase 3 训练阶段常见追问

1. 你如何证明训练流程是可复现的？
- 回答：我把流程固定为 `server init -> data check -> smoke -> full`，并把模型路径、关键超参、日志目录、checkpoint 路径写入文档；同样命令可在新机器复现。

2. 为什么要同时准备 in-domain 和 external 两套测试？
- 回答：只看 in-domain 容易高估效果；external 能检验跨分布泛化。两套同时报告，结论更稳健。

3. 你怎么控制 GPU 成本？
- 回答：先做短时 smoke，确认链路和显存后再开 full；每次只改少量超参并记录效果，避免反复长训试错。

4. 为什么你的 full 训练只有大约 65 秒，这还能算正式训练吗？
- 回答：可以。因为当前 v1 数据规模较小，所以 full 本身不会很长；但它已经真实完成了正式配置下的模型加载、训练、loss 记录、输出目录写出和 loss 曲线生成。对当前阶段来说，这已经足够证明 LoRA 微调闭环成立。

5. `train_loss = 1.6737` 为什么会高于中途打印的 `1.4035`？
- 回答：因为最终 `train_loss` 是整轮训练的聚合值，不等于最后一次单步 loss；中途日志更适合看趋势，最终指标更适合看整轮平均表现。两者不矛盾。

6. full 训练完成后，你下一步做什么？
- 回答：不是立刻继续加时长训练，而是先做 in-domain test 和 external_eval 的结果汇总，再分析成功样例和失败样例，确认模型到底学到了什么、没学到什么。

## 环境配置阶段常见追问

1. 为什么要自己建 conda 环境，而不是直接用平台默认环境？
- 回答：默认环境不可控，依赖冲突很难复现。我把训练环境独立到 `paper_ft`，并固定了 `torch/transformers/tokenizers/huggingface_hub/llamafactory` 版本，后续复现更稳。

2. 你真实遇到过什么环境问题？
- 回答：遇到过 `tmux` 缺失、模型路径错误、`huggingface_hub` 版本冲突、`torchvision::nms` 报错。我最后通过重建 conda 环境并统一 CUDA wheel 版本解决。

3. 为什么用无卡模式先配环境？
- 回答：环境安装和数据检查主要吃网络和磁盘，不一定需要 GPU；先用无卡模式处理这些准备工作，再切回 GPU 跑训练，成本更合理。

4. 为什么后来改用 ModelScope 下载模型？
- 回答：因为服务器到 `huggingface.co` 实测超时，但训练环境本身已经通过校验，所以我没有重建环境，而是切换模型下载源。这是网络层优化，不是训练方案变化。

5. 你怎么确认模型不是“看起来下载了”，而是真下载完整了？
- 回答：我做了文件级验收，不只看终端进度条；检查了配置文件、分词器配置、三个权重分片和总目录大小约 `7.6G`。

6. 为什么后来又升级到 `torch 2.6.0+cu124`？
- 回答：因为 Qwen3 首次训练不是卡在数据，而是在模型初始化时报兼容错误；说明问题在版本矩阵，不在样本或权重文件，所以我升级了 PyTorch 版本线，而不是误改数据或换模型。

7. 升级 `torch` 后为什么还要回钉 `numpy/fsspec/pillow`？
- 回答：因为 `pip` 会把未显式锁定的依赖解析成最新版本，这次把 `numpy/fsspec/pillow` 带偏后又和 `llamafactory/datasets/gradio` 冲突；所以我把它们回钉到项目兼容版本，并用 `pip check` 做最终验收。

8. 你怎么证明这不是“准备好了”，而是真的跑通了一次训练？
- 回答：因为我已经完成了一次真实 `smoke` 训练，记录到了训练时长、loss、吞吐、checkpoint 和输出目录；这说明链路不是停留在环境层，而是已经完成了真实训练闭环。

9. 现在你怎么证明不是只跑通了 smoke，而是正式训练也完成了？
- 回答：因为我已经完成了一次真实 `full` 训练，拿到了 `outputs/qwen_lora_v1_full/` 和 `training_loss.png`，并记录了训练时长、平均 loss、中间 loss 日志和正式输出目录。这说明项目已经从“训练可启动”推进到“正式训练已完成，待评测分析”。

10. 如果老师说“你这还是像在跑命令，不像在做项目”，你怎么回？
- 回答：我会把它定义成分阶段可复现的 LoRA 微调工程闭环，而不是一条命令。因为我保留了数据构建、清洗、切分、防泄漏、环境固定、模型验收、smoke、full 和日志复盘的完整过程，每一步都有脚本、验收标准和故障修复记录。
