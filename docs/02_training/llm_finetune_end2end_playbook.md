# LLM 微调闭环实战手册（End-to-End Playbook）

> 目标：让你不仅“跑通一次”，而是能完整讲清楚自己如何做了一个可复现的微调项目。

## 0. 你在做什么项目

- 题目：基于 Qwen 的中文论文精读与实验解析助手
- 路线：LLaMA-Factory + Qwen(2.5/3) + LoRA SFT
- 闭环定义：环境准备 -> 数据准备 -> 数据校验 -> 训练冒烟 -> 正式训练 -> 评测复盘 -> 面试表达
- 环境实操与真实踩坑记录见：`docs/02_training/environment_setup_guide.md`

---

## 1. 闭环全景图（7 个阶段）

1. 环境与执行链路
2. 任务定义与边界
3. 数据工程（准备/清洗/标注）
4. 训练配置与冒烟验证
5. 正式训练与日志追踪
6. 评测与案例分析
7. 简历与面试讲述

每一阶段都要回答 3 个问题：

- 我做了什么（What）
- 为什么这么做（Why）
- 结果是什么（Result）

---

## 2. 阶段 0：环境与执行链路

### 2.1 为什么要单独记录环境阶段

- 微调项目最容易浪费时间的地方，不是模型本身，而是环境和依赖冲突
- 如果环境不可复现，后面的训练结果也不可复现
- 老师追问时，这一段最能区分“会跑命令”和“理解工程链路”

### 2.2 这一阶段最小产出

- 可登录的训练服务器
- 固定的 conda 环境
- 已验证的一组依赖版本
- 已通过的数据检查命令

### 2.3 这阶段要回答的问题

- 为什么选 4090
- 为什么区分无卡模式和有卡模式
- 为什么要钉死依赖版本
- 遇到报错时如何判断是系统层、路径层还是 Python 依赖层问题

---

## 3. 阶段 1：任务定义与边界

### 2.1 业务任务拆分

本项目固定四类任务：

1. `contribution_extraction`：贡献提取
2. `method_comparison`：方法对比
3. `experiment_interpretation`：实验解析
4. `defense_followup`：答辩追问

### 2.2 为什么要拆成四类

- 防止数据只训练成“泛泛总结”
- 方便后续按任务类型评估短板
- 面试讲述更清晰：你不是做了单点问答，而是做了“能力矩阵”

---

## 4. 阶段 2：数据工程（你必须亲手做）

## 3.1 数据来源与结构

- 原始案例库：`data/raw/paper_cases_v1.json`
- 每条 case 包含：标题、问题、方法、基线、结果、局限

### 3.2 训练样本结构

每条 JSONL 样本字段：

- `id`: 唯一编号
- `task_type`: 四类任务之一
- `instruction`: 要模型完成什么任务
- `input`: 给模型的论文上下文
- `output`: 标准答案
- `source_case_id`: 可追溯来源

### 3.3 构建与校验命令

```bash
python scripts/build_dataset_v1.py
python scripts/check_dataset_v1.py
python scripts/export_dataset_readable.py
```

### 3.4 你要做的人工动作（最关键）

1. 打开 `train_v1_readable.md`，抽检 10 条
2. 找出 3 条质量一般的样本（不同任务类型）
3. 回到 `paper_cases_v1.json` 或脚本模板逻辑修改
4. 重新构建 + 校验 + 导出
5. 在 `progress_log.md` 记录“改了什么、为什么”

### 3.5 数据质量红线

- 不造数据：输出不能出现输入里没有的指标/结论
- 不空话：避免“效果很好”这类无信息表达
- 不偏科：train/val 四类任务都必须覆盖

---

## 5. 阶段 3：训练配置与冒烟

### 4.1 为什么先冒烟

- 先验证路径、依赖、显存、日志链路
- 失败成本低，避免长训练白烧钱

### 4.2 当前关键配置文件

- `configs/lora_sft_qwen25_3b_v1.yaml`
- `scripts/run_train_smoke.sh`
- `configs/lora_sft_qwen_v1_full.yaml`
- `scripts/run_train_full.sh`

### 4.3 冒烟成功标准

- 训练命令成功启动并结束
- 输出目录有日志和 checkpoint 产物
- 无路径错误、无 OOM、无数据字段错误

---

## 6. 阶段 4：正式训练（租 4090 后）

### 5.1 推荐流程

1. 跑 smoke
2. 看速度与显存占用
3. 调整超参后开正式训练

最小命令块：

```bash
export MODEL_PATH=/root/autodl-tmp/modelscope-cache/Qwen/Qwen3-4B
bash scripts/run_train_smoke.sh
bash scripts/run_train_full.sh
```

### 5.2 常调参数

- `per_device_train_batch_size`
- `gradient_accumulation_steps`
- `cutoff_len`
- `learning_rate`
- `num_train_epochs`

### 5.3 训练期间重点观察

- loss 是否稳定下降
- 是否出现过拟合迹象（train 好，val 差）
- 吞吐是否异常波动（机器状态问题）

---

## 7. 阶段 5：评测与案例

### 6.1 评测最小集（建议）

- 每类任务至少 10 条人工评测样本
- 维度：
  - 事实正确性
  - 结构清晰度
  - 对比解释力度
  - 追问一致性

### 6.2 案例整理

- 每类挑 2 条“明显优于 base”的案例
- 每类挑 1 条“失败案例”，写原因和改进方向

---

## 8. 阶段 6：面试讲述模板

## 7.1 30 秒版本

“我做了一个中文论文精读助手微调项目，拆成贡献提取、方法对比、实验解析和答辩追问四类任务。先做结构化案例库和可复现数据构建，再用 LoRA 在 Qwen 上训练，最后按任务类型评测并沉淀失败案例，形成了可复盘、可讲清楚的工程闭环。”

### 7.2 2 分钟版本（建议你背）

1. 目标：为什么做这个助手  
2. 数据：如何从 case 到 JSONL、如何做清洗与质检  
3. 训练：为什么 LoRA、怎么做 smoke 和正式训练  
4. 结果：哪些任务提升明显、哪里还不行  
5. 复盘：下一步怎么扩数据、怎么继续调参

---

## 9. 你今天就能执行的“学习闭环”

1. 手工修改 3 条样本（不同任务类型）
2. 重新构建 + 校验 + 导出可读版
3. 记录修改原因与效果
4. 用“2 分钟版本”讲一遍给我听（文字也行）

做到这 4 步，你就不是“会运行命令”，而是已经在做完整的微调工程。
