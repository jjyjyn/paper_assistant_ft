## A. 按天复盘

### Day 1：环境与链路

- 做了什么：
  - 本地/服务器仓库同步链路跑通
  - 训练环境初始化与依赖安装
  - 模型下载路径打通（含镜像策略）
- 为什么这样做：
  - 先解决可执行性，否则训练阶段会反复返工
- 可讲要点：
  - “先链路后训练”是工程效率最高的路线

### Day 2：数据集第一版

- 做了什么：
  - 统一四类任务 schema
  - 结构化案例库（20 case）+ 自动扩展为 80 条样本
  - 脚本化构建与校验（可复现）
- 为什么这样做：
  - 在有限时间内优先拿到“可训练 + 可追溯 + 可扩展”的数据底座
- 可讲要点：
  - 用模板起步保证速度，再用高难人工样本补泛化

### Day 3：训练前准备与环境固定

- 做了什么：
  - 在租用服务器上完成 conda 环境重建
  - 固定 `torch/torchvision/transformers/tokenizers/huggingface_hub/llamafactory` 版本
  - 在无卡模式下完成依赖安装与数据检查
  - 补齐 smoke/full 两套训练入口
- 为什么这样做：
  - 先把环境问题和训练问题拆开，否则一旦报错无法快速定位
  - 版本钉死后，后续 smoke/full 和复现实验都会更稳
- 可讲要点：
  - “先固环境，再开训练；先无卡装依赖，再有卡跑训练”是这次的核心工程策略

## B. 按主题复盘

### 1) 数据准备

- 原始数据形态：结构化 case（标题、问题、方法、基线、结果、局限）
- 目标数据形态：SFT JSONL（instruction/input/output）
- 任务类型：贡献提取、方法对比、实验解析、答辩追问

### 2) 数据清洗

- 字段完整性：检查 `id/task_type/instruction/input/output/source_case_id`
- 任务合法性：`task_type` 必须在固定集合内
- 空值与格式：过滤空字符串、非法 JSON 行
- 分布均衡：分层切分，保证 train/val 四类任务都有覆盖

### 3) 标注策略

- v1：模板生成 + 结构化输出（快速起步）
- v2：人工补充高难样本（冲突结果、边界问题、多轮追问）
- 质检流程：脚本校验 + 人工抽检（事实一致性、表达质量）

### 4) 显卡选择

- 选 4090 的原因：
  - 24GB 对 3B-4B LoRA 足够
  - 单小时成本低，适合快速试错
- 执行策略：
  - 先本地做完数据与脚本，再租机训练，避免空转计费

### 5) 环境与执行链路

- 镜像选择：
  - 选择 `Miniconda / Python 3.10 / Ubuntu 22.04`，避免直接依赖平台预装训练框架
- 依赖策略：
  - 不追求最新版，而是固定兼容版本组合
- 模式区分：
  - 无卡模式：装环境、拉代码、跑 check
  - GPU 模式：只负责训练
- 真实踩坑：
  - 缺 `tmux`
  - 模型路径写错
  - `huggingface_hub` 与 `transformers` 版本冲突
  - `torchvision::nms` 报错
- 处理原则：
  - 先判断问题属于系统层、路径层还是 Python 依赖层
  - 核心依赖链乱掉时直接重建环境，不在坏环境上持续打补丁

## C. 高频追问与答法

1. 为什么不直接手写几千条？
- 先跑闭环，再按误差扩容。可复现和迭代速度比“盲目堆量”更关键。

2. 你如何证明数据质量？
- 有脚本校验、分布检查、source_case_id 追溯，以及人工抽检记录。

3. 为什么要 smoke run？
- 先验证路径和超参，避免正式训练浪费 GPU 计费时长。

4. 为什么不直接用平台默认环境？
- 训练环境一旦不可控，报错时很难归因。我选择自建 conda 环境并钉死版本，保证后续 smoke/full 可复现。

5. 为什么要区分无卡模式和有卡模式？
- 因为依赖安装和数据检查不一定需要 GPU。先用无卡模式做准备，再把 GPU 时间留给训练，成本更合理。
