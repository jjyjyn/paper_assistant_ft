# 老师追问题库（按阶段拷打）

> 目标不是背答案，而是知道老师为什么这么问、你该从哪一层回答。

## 1. 项目定位与转向动机

### 问：你是网安专业，为什么做这个 AI 项目？

答：
- 我不是空泛地说“想转 AI”，而是选了一个完整微调项目，自己从数据准备、环境配置、训练到评测做一遍。
- 这个项目能证明我不是只会调用 API，而是理解模型训练链路和数据工程基本方法。
- 对老师来说，重点不是我原专业是什么，而是我有没有把 AI 项目做成可复现、可解释、可持续迭代的工程闭环。

### 问：为什么选“论文精读与实验解析助手”这个题目？

答：
- 这个题目天然适合拆成多种能力，而不是单一问答。
- 它同时覆盖信息提取、方法比较、实验解释、追问答辩四类任务，比较适合作为能力矩阵来做微调。
- 项目结果也更容易展示给老师看，因为输出是否清楚、是否贴近论文语境，很容易直观判断。

## 2. 任务定义与数据设计

### 问：为什么固定四类任务？

答：
- 如果只做单一 QA，模型很容易学成“泛泛总结”。
- 拆成四类任务后，可以分别检查贡献提取、对比解释、实验解读、追问一致性，便于定位短板。
- 这也让项目从“一个问答 demo”变成“一个能力矩阵微调项目”。

### 问：为什么不用真实论文全文直接训练？

答：
- 全文直接训练成本高，而且标注口径很难统一。
- 我先把论文关键信息抽象成结构化 case，再映射到四类 SFT 样本，先得到可控、可复现的数据底座。
- 后面如果要升级，可以在这个底座上继续扩展，而不是推翻重来。

### 问：为什么不是直接手写几千条？

答：
- 当前阶段目标是先做完整闭环，不是盲目追求样本数量。
- 我优先做“中等规模、结构清晰、可验证”的数据，再根据错误类型补高难样本，这比一开始堆量更有效。

## 3. 数据清洗与质量控制

### 问：你怎么保证 output 不是瞎编的？

答：
- 我要求 output 只能使用 input 里已经给出的事实，不允许新增指标、结论、基线。
- 这条规则一部分靠模板约束，一部分靠人工抽检，一旦发现问题，回到 raw case 修源头，而不是直接改 jsonl。

### 问：你怎么做清洗？

答：
- 清洗分四层：
- 第一层是字段完整性，必须有 `id/task_type/instruction/input/output/source_case_id`
- 第二层是任务合法性，`task_type` 只能来自四个固定集合
- 第三层是事实一致性，输出不能脱离输入
- 第四层是表达质量，尽量减少空话，保留结构化回答

### 问：你遇到过什么数据质量问题？

答：
- 典型问题有四类：
- 幻觉：output 引入 input 没有的指标
- 模板化过重：结构有了，但信息密度不够
- 数据泄漏：同一 case 跨 split
- 外部泛化不足：in-domain 好看，external 一般
- 我已经把这四类问题单独整理进数据清洗文档里，并给出对应修法。

## 4. 切分与防泄漏

### 问：为什么从 72/8 改成 64/8/8？

答：
- 早期只有 train/val，后面补了项目内测试集和 external_eval。
- 现在主数据是 case-level 的 `64/8/8`，外加独立 external_eval `32` 条样本。
- 这样训练、验证、同分布测试、跨分布测试各自职责更清楚。

### 问：为什么强调 case-level split？

答：
- 因为同一篇论文可以扇出四条任务样本。
- 如果按样本级随机切分，同一论文的不同任务很可能同时落到 train 和 val/test，评测会被污染。
- 所以我按 `source_case_id` 做切分，并在 check 脚本里校验跨 split 不允许重复。

### 问：为什么还要做 external_eval？

答：
- 只看 in-domain 结果，很容易高估模型能力。
- external_eval 强调跨数据分布泛化，能更真实地暴露“记模板”和“真理解”的差别。

## 5. 环境配置与工程执行

### 问：你是怎么配训练环境的？

答：
- 我在租用服务器上用 Miniconda 单独建了 `paper_ft` 环境，并把关键依赖版本固定。
- 不是依赖平台预装环境直接开训，因为那样冲突不可控。
- 我还区分了无卡模式和有卡模式：无卡装环境、拉代码、跑检查；有卡只负责训练。

### 问：为什么不一直用 GPU 模式装环境？

## 6. 训练策略

### 问：为什么先 smoke 再 full？

答：
- smoke 的目标不是拿指标，而是验证训练链路。
- 先确认路径、依赖、显存、日志都没问题，再开 full，能显著降低 GPU 浪费。

### 问：smoke 看什么，不看什么？

答：
- 看：能不能启动、会不会 OOM、日志是否完整、loss 是否正常下降
- 不看：最终效果好不好，因为 smoke 不是用来下最终结论的

### 问：你准备调哪些参数？

答：
- 第一批重点是 `per_device_train_batch_size`、`gradient_accumulation_steps`、`cutoff_len`
- 因为它们直接决定显存占用、吞吐和训练稳定性
- 学习率和 epoch 会在链路稳定后再调

## 10. 补充：无卡模式预下载与模型源选择

### 问：为什么改用 ModelScope 下载模型，而不是继续坚持 Hugging Face？

答：
- 原因不是训练框架变了，而是服务器到 `huggingface.co` 的网络路径实测超时。
- 我把问题分开处理：
- 训练环境本身是健康的，所以不应该为了下载问题去重建训练环境。
- 模型获取路径可以替换，所以改用 ModelScope 官方源完成下载。
- 这体现的是工程上的故障隔离，而不是拍脑袋换方案。

### 问：你怎么证明模型真的完整下载了？

答：
- 我没有只看 `successfully` 文本，而是做了文件级验收。
- 具体看了：
- `config.json`
- `tokenizer_config.json`
- 三个 `safetensors` 分片文件
- 总目录大小约 `7.6G`
- 这样即使终端进度条有残影，也不会误判状态。

### 问：为什么要先无卡下载，再 GPU 开机训练？

答：
- 下载模型主要消耗的是网络和磁盘，不是 GPU 算力。
- 如果用 GPU 模式下载，相当于把付费 GPU 时间浪费在等待网络上。
- 我把无卡模式用于环境、数据检查和模型预下载，把 GPU 模式专门留给 `smoke/full`，这是成本控制和工程节奏的结合。

### 问：为什么后来又升级了 PyTorch 到 2.6？

答：
- 因为 Qwen3 在首次 `smoke` 时不是卡在数据，也不是卡在权重下载，而是在模型初始化阶段报错。
- 我定位到这是 Qwen3 与旧版 PyTorch 的兼容问题，不是数据问题。
- 所以我没有改数据，也没有换模型，而是把 PyTorch 升到 `2.6.0+cu124`，让版本矩阵对齐到更稳的区间。

### 问：升级 PyTorch 后为什么又要手动回钉 `numpy/fsspec/pillow`？

答：
- 因为 `pip` 在强装大版本 wheel 时，只对我显式指定的包负责。
- `torch` 相关包装好了，但它的周边依赖会被解析成镜像里的最新版本。
- 这次就把 `numpy/fsspec/pillow` 带偏了，进而和 `llamafactory/datasets/gradio` 冲突。
- 所以正确做法不是重装一遍，而是把被带偏的包回钉到项目兼容版本。

## 12. 补充：评测阶段的老师追问

### 问：为什么你不用一个 accuracy 就结束，而要同时保留 `exact_match`、`char_f1` 和逐条报告？

答：
- 因为这个项目不是单标签分类，而是生成式结构化回答。
- 如果只看一个 accuracy，很容易把“表达不同但意思接近”和“真的答错”混在一起。
- 所以我把评测拆成三层：
  - `exact_match`：看格式和内容是否能严格对齐
  - `char_f1`：看预测和参考答案的大致重合程度
  - `report.md`：看具体样例，做人工抽检和失败归因
- 这样结果既能量化，也能解释。

### 问：为什么要同时跑 `test_v1` 和 `external_eval_v1`？

答：
- 因为只看项目内测试，很容易高估模型。
- `test_v1` 主要回答“模型对当前数据分布学得怎么样”。
- `external_eval_v1` 主要回答“模型离开当前分布后还能不能保持结构、事实和解释能力”。
- 这两组结果一起看，才能说明模型不是只记住了训练模板。

### 问：如果 `char_f1` 还可以，但 `exact_match` 很低，你怎么解释？

答：
- 这通常说明模型学到了部分内容，但格式、措辞或结构不够稳定。
- 对论文助手这种任务来说，这不是“完全失败”，但说明输出可控性还不够强。
- 后续我会重点补：
  - 模板边界样本
  - 格式稳定性样本
  - 多种表述风格下的同任务样本

### 问：如果 external 表现明显差于 test_v1，你下一步怎么做？

答：
- 我不会先盲目继续加训练轮数。
- 我会优先判断问题属于哪一类：
  - 数据分布过窄
  - 模板偏置过强
  - 某类任务样本难度不足
  - 输出结构在跨域场景下变差
- 然后优先补有代表性的外部失败样例，而不是只继续刷 loss。

### 问：如果你首轮评测里 `exact_match_rate` 是 0，你怎么向老师解释？

答：
- 我不会直接把它解释成“模型没学会”。
- 我会先排查三类问题：
  - 输出格式是否偏了
  - 推理过程是否泄露到最终答案
  - 评测脚本是否把无关内容也算进了打分
- 这次我的首轮评测真实发现就是：
  - 模型几乎普遍输出了 `<think>` 推理过程
  - 这直接污染了 `exact_match` 和 `char_f1`
- 所以这轮结果更像一次“评测故障定位”，而不是最终模型能力宣判。

### 问：你怎么证明这是评测污染，而不是模型完全不会？

答：
- 因为报告里不是空白输出，也不是胡言乱语。
- 更典型的情况是：
  - 模型先输出长段与任务相关的分析
  - 但没有稳定落到目标格式
- 比如本项目里：
  - `v1_0012` 本应输出三组 `Q/A`
  - 结果模型在 `<think>` 后只留下了 `以下是模拟答辩老师追问的三`
- 这说明它不是完全没方向，而是“答案通道被思维链污染并截断了”。

### 问：那你后面怎么修？

答：
- 我没有直接去改训练数据或继续加轮数。
- 我先修评测脚本：
  - 剥离 `<think> ... </think>`
  - 保留 `raw_prediction`
  - 再重跑 test 和 external
- 因为如果评测本身脏了，后面所有结论都会跟着脏。

## 13. 补充：更具体的失败样例追问

### 问：你能举一个具体失败样例吗？

答：
- 可以。`v1_0012` 这个样例要求输出 3 组 `Q/A`。
- 但模型先输出整段 `<think>`，最后真正答案区只剩“以下是模拟答辩老师追问的三”。
- 这说明问题不是完全没方向，而是推理泄露后把最终答案截断了。
- 这个例子让我确认，第一轮低分里至少有一部分是输出控制故障，而不只是知识掌握不足。

### 问：再举一个不是截断，而是格式不稳定的例子？

答：
- `v1_0062` 很典型。
- 这个任务要求做结构化方法对比，但模型没有直接输出对比结构，而是反复停留在“我现在需要帮助用户分析这个方法”这种自我分析句式里。
- 这说明模型已经捕捉到任务目标，但没有稳定收束到目标模板。
- 所以这里的主要问题是“格式和收束能力”，不是完全不理解论文内容。

### 问：你怎么理解 `exact_match_rate` 和 `avg_char_f1`？

答：
- `exact_match_rate` 是最严格的指标，要求模型输出和参考答案在字符串层面完全一致。
- 对结构化生成任务来说，这个指标很容易被格式差异、额外说明、`<think>` 泄露直接打成 0。
- `avg_char_f1` 更宽松，它衡量的是字符级重合程度，更适合看“内容有没有大体答到”。
- 所以：
  - `exact_match` 更像“最终答卷是否干净且格式完全对齐”
  - `char_f1` 更像“内容是否有部分学到”
- 这次首轮评测 `exact_match = 0` 但 `char_f1` 还有 `0.38~0.41`，就说明不能简单说模型完全不会，而要回到报告里看失败模式。
